# Allocation optimization is one of the highest ROI performance skills in Go.

Why?

```
Allocation = not free
Heap allocation = more expensive
More heap objects = more GC work
More GC = more latency / CPU
```

Senior engineers optimize by asking:

```
Can I avoid allocating at all?
If not, can I allocate less often?
If not, can I reuse memory?
```

# Why Allocations Matter

Every allocation may involve:

```
allocator bookkeeping
memory zeroing
pointer tracking
future GC scanning
cache misses
```

In high RPS systems:

```
10 allocs/request × 100k req/s = 1M allocs/s
```

That becomes real CPU cost.

# Stack vs Heap — Core Idea

Go has two common memory places:

## Stack
```
Fast
Per goroutine
Automatically freed when function returns
No GC needed
```

## Heap

```
Shared memory area
Lives beyond function scope if needed
Managed by GC
More expensive
```

# Best Case = Stack Allocation

Example:

```go
func f() {
    x := 10
    y := [3]int{1,2,3}
}
```

Usually:

```
x and y on stack
```

When function returns, memory gone automatically.

Very cheap.

# Heap Allocation Happens When Needed

If object may outlive current function:

```go
func makeUser() *User {
    u := User{Name: "A"}
    return &u
}
```

u likely goes to heap because pointer returned.

# Escape Analysis Decides

Go compiler uses:

```
Escape Analysis
```

Question:

```
Can object safely stay inside function lifetime?
```

If yes:
```
stack
```

If no:
```
heap
```

# Use Stack Instead of Heap — Practical Patterns

### Bad (forces heap often)

```go
func NewPoint() *Point {
    p := Point{1,2}
    return &p
}
```

Returns pointer.

### Better if pointer unnecessary

```go
func NewPoint() Point {
    return Point{1,2}
}
```

Return value.

```
Compiler can often keep on stack/registers.
```

# Small Struct Values Are Cheap

Many devs overuse pointers:

```
func process(u *User)
```

Sometimes better:

```
func process(u User)
```

Especially small structs.

```
Pointers may increase heap escapes and GC pointer scanning.
```

# Example: Pointer Causing Escape
```go
func main() {
    x := 10
    fmt.Println(&x)
}
```

Address escapes to interface call path often.

May move x to heap.

# Check With Compiler

Use:

```go
go build -gcflags="-m"
```

Output examples:

```
moved to heap: x
does not escape
```

Critical senior tool.

# Avoid Interface / Reflection in Hot Paths

**Sometimes values passed into interface{} can escape more easily.**

Example:

```
fmt.Println(x)
```

Can trigger escapes depending on context.

# Returning Slices Can Still Be Fine

Slice header returned by value:

```go
func build() []int
```

```
But backing array may be heap if needed beyond function.
```

That’s normal.

# Preallocate Slices — Why It Matters

Without preallocation:

```go
var nums []int
for i := 0; i < 1000; i++ {
    nums = append(nums, i)
}
```

Slice grows repeatedly.

Growth may allocate new larger arrays many times.
