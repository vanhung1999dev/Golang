# What Context Really Means

Context is **not the lifecycle of a function itself.**

It is the lifecycle of the work being performed by that function. <br>

That work might be:

- one HTTP request
- one DB transaction
- one background job
- one queue message
- one retry attempt
- one user session step
- one pipeline stage

So context = control object for work lifetime

# 1. Why Context Exists

```
Client request arrives
   ↓
API handler
   ↓
Call DB
Call Redis
Call 3rd-party API
Start worker goroutines
```

Now client disconnects.

Without context:

- DB query still runs
- HTTP call still waits
- Goroutines still alive
- Memory leaks
- CPU waste
- Connection exhaustion

So Go introduced context.Context.

# 2. What is Context Internally?

It is an interface:

```go
type Context interface {
    Deadline() (deadline time.Time, ok bool)
    Done() <-chan struct{}
    Err() error
    Value(key any) any
}
```

# 3. Meaning of Each Method

### A. Done()

Returns channel closed when canceled.

```go
select {
case <-ctx.Done():
    return
}
```

Used everywhere.

### B. Err()

Why canceled?

```go
context.Canceled
context.DeadlineExceeded
```

### C. Deadline()

Returns timeout time.

Useful for DB drivers, HTTP clients.

### D. Value()

Stores metadata.

Example:

```
requestID
userID
traceID
tenantID
```

# 4. Concrete Implementations Inside Runtime

### emptyCtx

Used by:

```
context.Background()
context.TODO()
```

Never canceled.

### cancelCtx

Supports manual cancellation.

Created by:

```
context.WithCancel(parent)
```

Contains:

```
type cancelCtx struct {
    Context

    mu       sync.Mutex
    done     chan struct{}
    children map[canceler]struct{}
    err      error
}
```

Important:

```
tracks child contexts
cancel cascades downward
```

### timerCtx

Used by:

```
context.WithTimeout()
context.WithDeadline()
```

Adds:

```
deadline time.Time
timer *time.Timer
```

### valueCtx

Used by:

```
context.WithValue()
```

Stores:

```
key
value
parent
```

**Linked-list lookup.**

# 5. Context Tree Model

```
Background()
   └── requestCtx
         ├── dbCtx
         ├── redisCtx
         └── apiCtx
```

If parent canceled:

```
requestCtx cancel
   ↓
dbCtx cancel
redisCtx cancel
apiCtx cancel
```

# Why Done Uses Closed Channel

Channel close is broadcast.

1000 goroutines waiting:

```
<-ctx.Done()
```

When close(done):

All wake immediately.

Cheap + elegant.

# 9.Timeout Propagation

```go
ctx, cancel := context.WithTimeout(parent, 2*time.Second)
```

```
If parent has earlier deadline:

child uses earlier one.
```

Example:
```
Parent: 1s
Child requests 5s
Actual child deadline = 1s
```

Because child cannot outlive parent.

# 10. Queue-Based Systems (VERY Important)
You asked queue systems. <br>

Imagine Kafka worker: <br>
```go
for msg := range queue {
    ctx, cancel := context.WithTimeout(context.Background(), 10*time.Second)

    process(ctx, msg)

    cancel()
}
```

Why?

- Each message gets bounded lifetime.

If stuck:

```
DB slow
third party hangs
deadlock
```

Timeout kills work.

# 11. Context in Worker Pools
```go
func worker(ctx context.Context, jobs <-chan Job) {
    for {
        select {
        case <-ctx.Done():
            return
        case job := <-jobs:
            process(ctx, job)
        }
    }
}
```

Shutdown system:

```
cancel()
```

All workers exit gracefully.
