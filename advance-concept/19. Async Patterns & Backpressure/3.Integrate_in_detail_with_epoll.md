# Core Rule
Context timeout does **NOT** magically stop a running goroutine mid-instruction.

It does:

```
1. Mark context canceled
2. Close ctx.Done()
3. Wake waiters
4. Libraries observe cancel and abort future/blocking operations
```

But if goroutine is already running on CPU:

it keeps running until:
- **reaches cancellation check**
- **enters cancellable syscall**
- **returns to runtime scheduling point**

# Case 1: Goroutine Is Reading User-Space Buffer (Already Got Data)

```go
n, _ := conn.Read(buf)

// now parsing bytes
json.Unmarshal(buf[:n], &req)
```

Then timeout fires.

### What Happens?

- ctx.Done closes
- but goroutine continues parsing

- **Because parsing is CPU work in user space.**

Context cannot instantly stop:

```
json.Unmarshal()
for loops
hashing
compression
CPU code
```

Until code checks context.

### Example

```go
for _, row := range rows {
    heavyCompute(row)
}
```

Timeout occurs mid-loop.

If no check:

```
loop continues
```

Better:

```go
for _, row := range rows {
    select {
    case <-ctx.Done():
        return ctx.Err()
    default:
    }

    heavyCompute(row)
}
```

# Case 2: Goroutine Blocked in Network Read Waiting for More Bytes

### Example:

```go
conn.Read(buf)
```

No data available.

Then timeout fires.

### What Happens?

Higher layer does:

```
close(fd)
set deadline now
shutdown socket
```

Kernel wakes epoll.

Runtime wakes goroutine.

Read returns:

```
i/o timeout
context deadline exceeded
```

# Case 3: Partial Read Already Happening in Kernel
Suppose kernel already copied some bytes.

```
NIC -> kernel socket buffer
kernel -> user buf copy started
```

Timeout fires during that. <br>

Usually

```
read syscall finishes first
returns bytes read
```

**Then next operation sees cancellation.**

**Because kernel copy is tiny/fast.**

# Case 4: Goroutine Busy CPU + Never Checks Context

```go
for {
    x++
}
```

Timeout fires.

What happens?

```
ctx canceled
goroutine keeps burning CPU forever
```

**Because context is cooperative.**
