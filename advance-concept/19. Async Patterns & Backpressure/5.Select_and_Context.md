# Go Context + select Deep Dive

## 1. What Does "Sharing Same Context" Mean?

When multiple functions or goroutines receive the **same context instance**, they all reference the same cancellation state.

```go
ctx, cancel := context.WithCancel(context.Background())

go worker1(ctx)
go worker2(ctx)
go worker3(ctx)
```

All workers use the same `ctx`.

That means they share:

* same `Done()` channel
  n- same cancel signal
* same deadline (if any)
* same values stored in that context

When `cancel()` is called, every goroutine waiting on `ctx.Done()` can wake up.

---

## 2. What If Contexts Are Different?

```go
ctx1, cancel1 := context.WithCancel(context.Background())
ctx2, cancel2 := context.WithCancel(context.Background())

go worker1(ctx1)
go worker2(ctx2)
```

These are separate contexts.

* `cancel1()` only affects `ctx1`
* `cancel2()` only affects `ctx2`

Each has its own `Done()` channel.

---

## 3. Is `ctx.Done()` a Shared Channel?

For the **same context**, yes.

```go
ch1 := ctx.Done()
ch2 := ctx.Done()
```

Both refer to the same underlying channel.

So if multiple goroutines do:

```go
<-ctx.Done()
```

They are all waiting on the same signal.

When canceled, that channel is closed.

Closing a channel wakes all receivers.

---

## 4. How Does `select` Wait List Work?

Example:

```go
select {
case <-ctx.Done():
    return
case msg := <-jobs:
    fmt.Println(msg)
}
```

When execution reaches `select`:

### Step A: Check Cases Immediately

Go checks whether any case is ready:

* Is `ctx.Done()` closed / readable?
* Is `jobs` ready to receive?

### Step B: If One Is Ready

One ready case is chosen immediately.

### Step C: If None Are Ready

The goroutine is parked (blocked).

Internally, the runtime places the goroutine into waiting queues for those channels.

Later:

* If context is canceled -> `Done()` channel closes -> wake goroutine
* If a job arrives -> `jobs` channel becomes ready -> wake goroutine

---

## 5. If Function Already Finished Running, How Can `select` Trigger?

It cannot.

If a function already returned, the goroutine is done.

```go
func worker(ctx context.Context) {
    fmt.Println("done")
}
```

After return:

* no running goroutine
* no waiting `select`
* no stack frame for that function

Calling `cancel()` later does nothing to that finished function.

---

## 6. Important Difference: Alive vs Finished

### Alive and Waiting

```go
func worker(ctx context.Context) {
    select {
    case <-ctx.Done():
        return
    }
}
```

This goroutine is alive and blocked. Canceling context wakes it.

### Finished

```go
func worker(ctx context.Context) {
    fmt.Println("done")
    return
}
```

This goroutine is gone. Nothing to wake.

---

## 7. Busy CPU Work Does Not Instantly Stop

```go
func worker(ctx context.Context) {
    heavyLoop()

    select {
    case <-ctx.Done():
        return
    default:
    }
}
```

If canceled during `heavyLoop()`, the context is canceled immediately, but the function reacts only after it reaches the next context check.

---

## 8. Mental Model

* `context.Context` = cancellation signal + metadata
* `ctx.Done()` = broadcast channel
* `select` = wait on one or more channels
* `cancel()` = close done channel
* finished goroutine = nothing left to trigger

---

## 9. Summary Table

| Situation                   | Result                           |
| --------------------------- | -------------------------------- |
| Many funcs use same ctx     | all can wake on cancel           |
| Different ctx objects       | only matching context wakes      |
| Goroutine blocked in select | wakes when case becomes ready    |
| Function already returned   | nothing happens                  |
| CPU work without checks     | stops later when code checks ctx |
