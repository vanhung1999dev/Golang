# Mental Model

Traditional thread-per-request systems:

```
100k tasks => 100k threads impossible
```


Go model:

```
100k tasks => 100k goroutines
maybe only 8 OS threads running
```

How?
```
Go runtime multiplexes goroutines onto OS threads.
```

That means:

```
Most “switches” happen in user space, not kernel.
```

# Two Types of Context Switches in Go

## 1. OS Context Switch (Expensive)

Kernel pauses one thread, runs another. <br>

Save/restore:
- CPU registers
- stack pointer
- instruction pointer
- TLS- 
- scheduling state
- memory mappings/cache disruption

Cost:

```
~1000ns to several µs+
```

Depending CPU/cache/load.

## 2. Goroutine Switch (Cheap)

Go runtime switches:

```
G1 -> G2
```

- On same thread.
- No kernel scheduler involved (usually).

Need mostly:

- save Go registers
- save stack pointer
- save program counter
- change active goroutine metadata

Cost often much lower.
```
Tens to hundreds ns range (varies)
```

## Why This Matters

If you build high concurrency services:
```
1M requests
```

Using threads:
```
Kernel thrashes with thread scheduling.
```

Using Go:
```
Runtime cheaply parks/wakes goroutines.
```

## What Is User-Space Scheduling?
Go runtime has its own scheduler:

```
G = goroutine
M = machine (OS thread)
P = processor token
```

A goroutine runs only when:

```
G attached to M through P
```

Runtime decides:

- which goroutine runs next
- when to block one
- where to wake one
- whether to spawn thread

Kernel only sees a few threads.

## Cheap Context Switching in Go

Because switching goroutines often means:

- same OS thread
- same address space
- same kernel thread
- same process

So avoid:

- kernel mode transition
- full thread scheduler involvement
- TLB/cache disruption (less severe)
- syscall overhead

### Example: Channel Blocking

```go
ch := make(chan int)

go func() {
    x := <-ch
}()
```
Receiver waits. <br>

Runtime does: <br>

```
G1 status = waiting
remove from run queue
pick next runnable goroutine
```

No kernel switch required if same thread has other runnable G. <br>

### Example: Mutex Contention

```go
mu.Lock()
```

If unavailable: <br>

Runtime parks goroutine: <br>

```
current G sleeps
scheduler runs another G
```

Again cheaper than blocking OS thread (in many cases).

## What Actually Saved During Goroutine Switch?

Inside runtime (gobuf concept): <br>

Stores roughly: <br>

```
sp  = stack pointer
pc  = program counter
g   = goroutine ptr
ctxt
bp (arch dependent)
```

Think:

```
pause function mid-flight, resume later
```

<br>

# Preemption in Go Big Picture


Async preemption is not:
```
Kill goroutine immediately
```

It is:
```
Request goroutine to yield CPU so scheduler can run others
```

## 1. How Go Detects “Running Too Long”

Each P has scheduler tick timing. <br>

Runtime tracks when a goroutine starts running. <nr>

Conceptually:

```
G started at T0
now = T1
if T1 - T0 > time slice (~10ms-ish)
   request preemption
```

Not exact wall-clock guarantee, but mental model is correct.

### Where Checked?

Inside scheduler/sysmon. <br>

There is a background monitor thread: <br>
```
sysmon
```

It periodically wakes and inspects runtime state: <br>

- stuck threads
- timers
- GC needs
- long-running goroutines

If an M/P runs same G too long:
```
sysmon marks it preemptible
```

## 2. How Go Marks a Goroutine for Preemption

Runtime sets flags like:

```
gp.preempt = true
```

And often:
```
stackguard0 = stackPreempt
```

This is clever. <br>

Normally stackguard0 protects stack overflow checks. <br>

Go reuses it to force a trap at next stack check. <br>

## 3. But What If Tight Loop Has No Function Calls?

```
for {
    x++
}
```

No call = no stack check. <br>

Older Go had issue here. <br>

Go 1.14 async preemption fixed this. <br>

## Real Mental Model
```
G1 runs long
sysmon notices
thread gets signal
if safe:
   G1 paused
   G1 back queue
   G2 runs
later G1 resumes
repeat as needed
```
