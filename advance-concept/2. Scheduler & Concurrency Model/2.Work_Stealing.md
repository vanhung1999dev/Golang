## Local Queue (fast path)
Each P has:

```
P.runq
```

Most goroutines are pushed/popped here.

#### Advantages:

- lock-free / low contention
- CPU cache locality
- scales across cores


### Local Queue Structure

Each P has circular buffer:
```
runq[256]   // roughly
head
tail
```

(Exact size may vary by version, but concept same.)

Operations:

- owner P push/pop mostly lock-free
- thieves steal half

<br>

## Global Queue (shared path)

Used when:

- local queue overflows
- fairness rebalancing
- no local work
- newly awakened goroutines in some paths
- handoff situations

Think:

- Global queue = overflow + balancing center

### Structure Internally

Simplified:

```
type schedt struct {
    runqhead G list
    runqtail G list
    lock mutex
}
```

There is:

```
runtime.sched.runq
```

Protected by lock.

<br>


# 🔹 Work Stealing

When a P is idle:

- steal from another P

Algorithm:
- Pick random P
- Steal half of its queue

### Why half?
- balance load without starving victim

### 🔥 Example
```
P1: [G1, G2, G3, G4]
P2: []

P2 steals → [G3, G4]
```

### 🧠 CPU insight
- reduces global contention
- improves cache locality
- minimizes synchronization

<br>

## When Goroutine Goes To Local Queue?

Most common:

```go
go func() {}
```

Current P usually pushes new goroutine into its own local queue. <br>

Why? <br>

Because creator goroutine likely related data in same cache. <br>

Good locality. <br>

# When Goes To Global Queue?

## 1. Local Queue Full
If P local queue full:
```
spill some goroutines to global queue
```

Instead of blocking creation.

#### Example

```go
for i:=0; i<1_000_000; i++ {
   go work()
}
```

Local queue cannot hold all. <br>

Overflow goes global. <br>

## 2. Fairness

If scheduler only used local queues: <br>

```
Busy P may monopolize own queue forever
```

So periodically runtime checks global queue. <br>

Even if local queue has work. <br>

This prevents starvation. <br>

## 3. Waking blocked goroutines

When goroutine wakes from:

- channel
- mutex
- syscall return
- netpoll
- timers

Sometimes inserted local, sometimes global depending context.<br>

## 4. Idle P Needs Work

If P local queue empty:

scheduler tries:

```
1. global queue
2. steal from another P
3. netpoll
4. timers
5. sleep
```

## Scheduling Search Order (Important)

When M with P wants runnable goroutine:

```
findRunnable():
    check local runq
    occasionally check global runq
    check netpoll
    steal from other Ps
    global again
    sleep
```

# Why Not Always Global First?

Because global queue requires lock. <br>

Local queue: <br>

- faster
- less contention
- better locality

So local preferred.

## Fairness Rule (Very Important Interview Topic)

Go runtime periodically pulls from global queue even if local has tasks. <br>

Why? <br>

Without this: <br>

```
P0 keeps spawning children into local queue forever
P1 idle
global tasks starve
```

## Work Stealing vs Global Queue
Two ways empty P gets work: <br>

### Global Queue

Take some runnable goroutines from shared queue.

### Stealing

Steal half from another P’s local queue. <br>

Why steal half? <br>

```
Because reduces repeated stealing overhead.
```

Example:
```
P1 has 100 tasks
P2 empty

P2 steals 50
```
Balances load quickly.
