## 🔹 G = Goroutine

A goroutine is NOT a thread. It is:

```
A user-space task managed entirely by Go runtime
```

### Internal structure (g)

Simplified: <br>

```
type g struct {
    stack       stack   // stack memory
    sched       gobuf   // saved registers (PC, SP)
    status      uint32  // Grunnable, Grunning, Gwaiting
    m           *m      // current thread
    waitreason  string
}
```

#### Stack (IMPORTANT)
- Starts small (~2KB)
- grows/shrinks dynamically

👉 unlike OS threads (fixed ~1MB)

#### sched (context)

Stores:

```
PC (program counter)
SP (stack pointer)
registers
```

👉 used for context switching


<br>

## 🔹 M = Machine (OS Thread)

Represents:

```
A real OS thread (pthread)
```

#### What M does:
- Executes goroutines
- Performs syscalls
- interacts with OS scheduler

#### Key constraint:
- **One M runs ONE goroutine at a time**

<br>

## 🔹 P = Processor (MOST IMPORTANT)

This is the secret sauce.

#### What is P?

```
A logical CPU token required to run Go code
```

### Contains:

```
type p struct {
    runq [256]g   // local run queue
    runqhead
    runqtail
    mcache        // memory cache
}
```

### Key rule:

```
M must acquire a P to execute Go code
```


### 🔥 Why P exists?

#### Without P:

- threads would compete globally
- high contention

#### With P:

- each P has local run queue
- reduces lock contention
- enables work stealing
