# How Stack Growth Happens

Every function call may include a stack check inserted by compiler. <br>

Conceptually:

```
if SP < stackguard0 {
    morestack()
}
call fn()
```

Meaning: <br>

Before entering function: <br>

```
Do I have enough room for next frame?
```

If no:

```
runtime.morestack()
```

### Example
```go
func A() { B() }
func B() { C() }
func C() { ...large locals... }
```

When entering C, runtime may detect insufficient room. <br>

Then: <br>

```
grow stack before C executes
```

# How Growth Actually Works (Modern Go)
Not page-by-page expansion. <br>

Instead: <br>

### Step 1: Allocate Bigger Stack

```
old = 2KB
new = 4KB
```

Usually doubles geometrically. <br>

### Step 2: Copy Old Stack Frames

Copy active stack contents to new region.

### Step 3: Fix Pointers

Adjust internal pointers that referenced old stack addresses.

### Step 4: Update Goroutine Metadata
g.stack.lo / hi now point new stack

### Step 5: Resume Execution

Program continues.

## Why Doubling?

Geometric growth gives amortized efficiency.

```
2KB -> 4KB -> 8KB -> 16KB -> ...
```

Avoids constant reallocations.

## Cost of Stack Growth
Growth is not free. <br>

Includes:

- allocation
- memcpy of active stack
- pointer adjustment
- possible cache misses

So if repeated often in hot path:

# Stack Shrink

## When Shrink Happens
Typically during garbage collection / stack scanning phases. <br>

Runtime inspects usage: <br>

```
actual live stack small
allocated stack large
```

Then may allocate smaller stack and copy down.

## Why Not Shrink Constantly?
Too much churn. <br>

If runtime shrank immediately after every call: <br>

grow/shrink/grow/shrink thrashing <br>

So shrinking is conservative. <br>
