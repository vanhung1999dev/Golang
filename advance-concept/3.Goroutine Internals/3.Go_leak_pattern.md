A goroutine leak means:

```
A goroutine is still alive, blocked, or looping forever,
even though it no longer does useful work.
```

Unlike memory leaks in C, Go GC can free memory only if unreachable.
But a leaked goroutine is **still reachable by runtime scheduler**, so it stays alive.

hat means leaks cause: <br>

- memory growth (stack + objects referenced by stack)
- scheduler overhead
- file/socket leaks
- DB connection exhaustion
- CPU waste
- latency spikes
- eventual OOM/cras

## Why Goroutine Leaks Are Dangerous
Each goroutine may have: <br>

- stack (~2KB+ grows dynamically)
- channel refs
- request context refs
- DB/socket handles
- timers
- heap objects kept alive

So:

- 100 leaked goroutines = annoying
- 100k leaked goroutines = outage

# Core Mental Model


A goroutine exits only when:

- function returns
- panic crashes it (unless recovered elsewhere)
- process exits
- runtime fatal error

There is no external kill goroutine API.

So if you start one: <br>

```go
go worker()
```

You must design how it stops.

## Leak Pattern #1: Blocked Receive on Channel

```go
func worker(ch <-chan int) {
    go func() {
        v := <-ch
        fmt.Println(v)
    }()
}
```

If nobody sends:

```
goroutine blocks forever on receive
```

### Fix

Use:

- close channel
- context cancel
- timeout
- select

```go
select {
case v := <-ch:
case <-ctx.Done():
    return
}
```


## Leak Pattern #2: Blocked Send on Channel

```go
func produce(ch chan<- int) {
    go func() {
        ch <- 1
    }()
}
```

If no receiver: <br>

```
goroutine stuck forever on send
```

Especially common with unbuffered channels.


### Fix

Use non-blocking send:

```go
select {
case logCh <- msg:
default:
}
```

Or bounded queue with metrics.

## Leak Pattern #3: Forgotten Background Worker

```go
func Start() {
    go func() {
        for {
            doWork()
            time.Sleep(time.Second)
        }
    }()
}
```

No stop path. <br>

When service reloads/tests run repeatedly: <br>

```
multiple zombie workers remain
```

### Fix
```go
func Start(ctx context.Context) {
    go func() {
        ticker := time.NewTicker(time.Second)
        defer ticker.Stop()

        for {
            select {
            case <-ticker.C:
                doWork()
            case <-ctx.Done():
                return
            }
        }
    }()
}
```

## Leak Pattern #4: HTTP Calls Without Timeout
```go
go func() {
    http.Get(url)
}()
```

If remote hangs:

```
goroutine blocked in network I/O long time

Many requests => leak-like resource pileup.
```

#### Fix

Use context timeout:

```go
ctx, cancel := context.WithTimeout(ctx, 3*time.Second)
defer cancel()
req = req.WithContext(ctx)
client.Do(req)
```

## Leak Pattern #5: Worker Pool Never Drained

```go
jobs := make(chan Job)

for i := 0; i < 10; i++ {
    go worker(jobs)
}
```

Workers:

```go
for job := range jobs { ... }
```

If jobs never closed during shutdown: <br>

```
workers wait forever
```

#### Fix

Close channel on shutdown or use context.

## Leak Pattern #6: Fan-Out Without Cancellation

```go
for _, item := range items {
    go fetch(item)
}
```

If request canceled after first success: <br>

others continue uselessly. <br>

Leak-ish burst. <br>

#### Fix
```go
Use errgroup.WithContext().

g, ctx := errgroup.WithContext(ctx)
```
Cancel siblings on first error/success depending design.

## Leak Pattern #7: time.After in Loops

```go
for {
    select {
    case <-work:
    case <-time.After(time.Second):
    }
}
```

Creates new timer each loop. <br>

Under load: <br>

timer objects accumulate until fired <br>

Not classic goroutine leak, but common runtime resource leak. <br>

#### Fix

Use reusable ticker/timer.

```go
ticker := time.NewTicker(time.Second)
defer ticker.Stop()
```

## Leak Pattern #8: Context Ignored

```go
go func() {
    for {
        process()
    }
}()
```

Request ends, but goroutine still running.

#### Fix

Always thread context through long-lived operations.

## Leak Pattern #9: Retry Loop Forever
```go
go func() {
    for {
        err := call()
        if err == nil { return }
    }
}()
```

No backoff, no max retry, no cancel.

Result: <br>

- infinite goroutine
- CPU burn
- thundering herd
