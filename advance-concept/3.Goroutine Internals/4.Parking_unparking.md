# Core Idea
When a goroutine cannot continue: <br>

- waiting for channel send/recv
- mutex unavailable
- select waiting
- timer sleep
- network I/O pending
- syscall coordination
- GC stop points

Go runtime usually does:

```
Park the goroutine
```

Meaning:

- Remove it from CPU scheduling.
- Mark it waiting.
- Run another goroutine.

Later when condition becomes true:

```
Unpark it
```

Meaning:

- Mark runnable again.
- Put on queue.
- Eventually resume.

## High-Level Lifecycle of a Goroutine

```
new -> runnable -> running -> waiting -> runnable -> running -> dead
```

Real Goroutine States (runtime)

```
Internally statuses like:

_Grunnable
_Grunning
_Gwaiting
_Gsyscall
_Gdead
_Gcopystack
_Gpreempted
```

### What Happens During Parking

Suppose G1 waiting on channel receive.

```go
x := <-ch
```

No sender exists. <br>

Runtime does roughly: <br>

```
1. Set G1 state = waiting
2. Link G1 into channel wait queue
3. Save execution context
4. Remove G1 from running
5. Call scheduler
6. Run another goroutine
```

So current OS thread does not waste time.

### Unparking = goready()

When sender arrives:

```go
ch <- 10
```

Runtime finds waiting receiver G1. <br>

Then:

```
1. Move data directly if possible
2. Change G1 waiting -> runnable
3. Put G1 on queue
```

Primitive often: <br>

```
runtime.goready(G1)
```

## findRunnable() Deep Dive

Scheduler core path conceptually:

```
findRunnable():
    1. check local runq
    2. check runnext
    3. occasionally global queue
    4. poll network events
    5. steal from other Ps
    6. timers ready?
    7. if none, idle/sleep
```
