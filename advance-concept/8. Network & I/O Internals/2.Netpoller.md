# High-Level Lifecycle

```
goroutine does conn.Read()

no data ready

=> goroutine parks
=> runtime registers fd with epoll
=> M can run another goroutine

later socket readable

=> epoll wakes runtime
=> goroutine becomes runnable
=> scheduler runs it again
```

# Create goroutine

### Create Goroutine

```go
go worker()
```

Compiler converts to runtime call:

```
newproc()
```

Runtime:

```
allocate G struct
initialize stack
set start function
status = runnable
push into current P local runq
```

### Result
```
P0 local queue:
[Gnew]
```

No thread created here. <br>

This is critical. <br>

# Scheduler loop

Each M runs:

```
schedule()
```
Pseudo:

```go
for {
    gp := findRunnable()
    execute(gp)
}
```


### findRunnable() Priority

Roughly:

- local runq
- global runq
- netpoller ready goroutines
- steal from other P
- idle / block

### Example
P0 local queue has G1 <br>

Then run G1 immediately. <br>

# Goroutine calls conn.Read()

Now important part. <br>

You run: <br>

```go
n, err := conn.Read(buf)
```

Go net package eventually enters runtime poll descriptor.

## Runtime Checks Socket

Socket FD is nonblocking. <br>
 
Attempts OS read: <br>

read(fd) <br>

### Case 1: Data already ready

- Kernel socket buffer has bytes.

- read returns immediately.

- Goroutine continues.

- No park.

### Case 2: No data

OS returns:

```
EAGAIN
```

Means:

- nothing available now

- Now runtime uses netpoll.

# What Happens Next

## Register interest

Runtime associates goroutine with fd waiting for READ. <br>

Internal pollDesc roughly: <br>

```
fd=12
rg = waiting goroutine G7
wg = nil
```

## Goroutine parks
```
gopark()
```

Meaning:

- current G status = waiting
- removed from CPU
- stack preserved

### Important Answer To Your Earlier Confusion

When goroutine parks: <br>

It is NOT in runq anymore. <br>

It leaves runnable queues. <br>

**Now waiting in pollDesc / timer / channel / mutex etc.**
