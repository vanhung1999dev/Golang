# Go Netpoller, Epoll/Kqueue, Goroutine Parking & Non-Blocking I/O Deep Dive

## 1. Overview

Go lets developers write code in a simple blocking style:

```go
n, err := conn.Read(buf)
```

But internally, the runtime uses:

* Non-blocking sockets
* Netpoller (`epoll` on Linux, `kqueue` on macOS/BSD)
* G/M/P scheduler
* Goroutine park / unpark

This gives readable code with high scalability.

---

## 2. Blocking vs Non-Blocking I/O

### Blocking Socket

```c
read(fd, buf, 1024)
```

If no data is available:

* Kernel puts the OS thread to sleep
* Thread is blocked until data arrives

### Non-Blocking Socket

Same file descriptor, but socket has `O_NONBLOCK` enabled.

If no data is available:

```text
read(fd) => EAGAIN
```

The thread is not blocked.

> Non-blocking mode changes syscall behavior. It does not remove the FD.

---

## 3. What Is a File Descriptor?

```text
fd=7 -> socket -> recv buffer / send buffer / TCP state
```

The FD exists as long as the socket is open.

---

## 4. Case 1 vs Case 2 for conn.Read()

### Case 1: Data Already Available

Client already sent bytes before Go calls `Read()`.

```text
kernel recv buffer = [data]
```

`read(fd)` succeeds immediately.

### Case 2: No Data Yet

```text
kernel recv buffer = []
```

`read(fd)` returns `EAGAIN`.

Runtime then:

* Parks goroutine
* Waits through epoll/kqueue
* Wakes goroutine when data arrives
* Retries read

## 5. G / M / P Model

### G = Goroutine

* Small stack
* Execution context
* Status

### M = Machine

* Real OS thread

### P = Processor

* Required to run Go code
* Owns local run queue

## 8. Why Deadlock Still Happens

Queues store waiters but do not guarantee progress.

```go
ch := make(chan int)
<-ch
```

No sender exists, so runtime panics:

```text
fatal error: all goroutines are asleep - deadlock!
```

---

## 9. How Runtime Detects Deadlock

Runtime checks:

* Runnable goroutines?
* Running goroutines?
* Pending timers?
* Netpoll wakeups possible?
* Syscall thread may return?

If none exist => deadlock.

---

## 10. Netpoller

Linux:

```text
epoll_wait()
```

macOS/BSD:

```text
kqueue()
```

When FD becomes ready:

* Find waiting goroutine
* `goready()`
* Put back to run queue

---

## 11. Full conn.Read() Flow

```text
conn.Read()
 -> read(fd)
    -> success => continue
    -> EAGAIN => park goroutine
                 run another goroutine
                 epoll waits
                 wake goroutine
                 retry read
```
