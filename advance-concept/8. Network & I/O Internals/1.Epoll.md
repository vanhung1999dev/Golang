# 1. Why epoll / kqueue exist

Imagine a server with:

- 100,000 TCP connections
- Most are idle
- Some occasionally send data

If you create 1 thread per connection, you waste memory + context switching.

#### Instead:

Use 1 or few threads <br>
Ask OS: <br>
“Tell me which sockets are ready” <br>

That is event-driven I/O. <br>

# 2. Before epoll: select() and poll()

## select()

Old API:

```
select(...)
```

### Problems:

- limited FD count
- copies FD sets user ↔ kernel each call
- scans all FDs every time: O(n)

If 100k sockets, terrible.

## poll()

Better than select:

```
poll(...)
```

- No FD limit issue, but still:
- pass all FDs every call
- kernel scans all FDs

Still O(n)

# 3. epoll solved this (Linux)

Linux introduced:

```
epoll_create()
epoll_ctl()
epoll_wait()
```

### Idea:

#### Separate:

Interest list
```
“What sockets I care about”
```

Stored once in kernel. <br>

Ready list

```
“What sockets are currently ready”
```

Kernel fills automatically. <br>

So no rescanning 100k FDs every loop.

# 4. Core Mental Model

```
Application:
   register socket fd=10 for READ

Kernel epoll:
   watching fd=10

NIC receives packet
→ socket receive buffer now has bytes
→ socket becomes readable
→ kernel pushes fd=10 into ready queue

app calls epoll_wait()
→ gets fd=10 instantly
```

# 5. File Descriptor first (critical)
Everything in Linux is file descriptor:

- socket
- file
- pipe
- eventfd
- timerfd

FD is integer:
```
3,4,5,6...
```

Per-process table:
```
fd 5 -> socket object in kernel
```
epoll watches these FDs.

# 6. What happens when socket receives packet
Step by step:

#### Client sends TCP packet

NIC receives.

#### Interrupt / softirq

Kernel networking stack processes packet.

#### Finds destination socket

Using:

dst IP
dst port
src tuple

#### Put bytes into socket receive buffer

Kernel socket struct has recv queue.

#### Mark readable

Now app can read.

#### Wake epoll waiters

If socket belongs to epoll interest list, kernel adds event.

# Level Triggered vs Edge Triggered

## Level Triggered (default)

If data still exists, keeps notifying. <br>

Example: <br>

socket has 100 bytes. <br>

You read only 20 bytes. <br>

Next epoll_wait() still returns readable. <br>

Easy mode. <br>

## Edge Triggered (EPOLLET)

Notify only when state changes: <br>

```
not readable -> readable
```

If you read only 20 of 100 bytes, remaining 80 may NOT notify again. <br>

Need drain until EAGAIN. <br>

Use nonblocking sockets. <br>

## Correct ET pattern
```
while (read(...) > 0) { ... }

if errno == EAGAIN stop
```

# Nonblocking sockets
Essential. <br>

Without it: <br>

```
read(fd)
```

may block thread. <br>

With nonblocking: <br>

```
if no data -> EAGAIN
```

Then continue processing others.

# Example Event Loop
```
while (1) {
   n = epoll_wait(epfd, events, ...)

   for each event:
      if listen socket:
          accept new conn
      else if readable:
          read()
      else if writable:
          write()
}
```

# How Go uses epoll

Go runtime netpoller: <br>

Linux: <br>

epoll_wait <br>

When goroutine does socket read: <br>

```
if no data:
goroutine parks
runtime keeps thread free
epoll waits
when ready:
goroutine runnable again
```

This is why Go handles many connections with few threads.
