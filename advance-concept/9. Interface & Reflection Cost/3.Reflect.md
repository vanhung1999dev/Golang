# Reflection means runtime inspecting types dynamically:
```go
reflect.TypeOf(x)
reflect.ValueOf(x)
FieldByName("UserID")
Call(...)
Set(...)
```

This is expensive for several reasons.

## Reason #1 — Metadata Traversal

Reflection walks runtime type descriptors:

```
struct fields
tags
offsets
method tables
embedded fields
visibility rules
```

This means pointer chasing in memory.

CPU hates:

```
random memory access
```

## Reason #2 — String Lookup

```
FieldByName("Email")
```

Needs searching fields by string.

Costs:

```
compare strings
iterate fields
possibly recurse embedded structs
```

Compared to compile-time:

```
user.Email
```

which is fixed offset load.

## Reason #3 — reflect.Value Wrapper

Reflection uses boxed container:

```
reflect.Value
```

It stores metadata + pointer + flags.

Every operation:

```
check kind
check settable
check addressable
convert pointer
```

Lots of branches.

## Reason #4 — No Inlining / Poor Optimization

Compiler can optimize:

```
u.Email = "a"
```

But cannot optimize:

```
rv.Field(i).SetString("a")
```

Runtime opaque behavior.

## Reason #5 — Allocations

Many reflection-heavy operations allocate:

```
[]reflect.Value
temporary objects
converted strings
interface wrapping
maps/slices
```

Then GC load increases.


## Example Real Cost
JSON libraries:

Old style:
```
encoding/json uses reflection heavily
```

Fast libraries:

```
easyjson
sonic
codegen serializers
```

Avoid reflection and can be much faster.

## Why Frameworks Use Reflection Anyway

Because developer productivity.

Example:

```
BindJSON(req)
ORM.Scan(&user)
DI container resolve()
tag parsing
router auto bind params
```

Tradeoff:

```
developer speed vs runtime speed
```

# Senior Rule: Reflection in Cold Paths Only

Good places:

```
startup initialization
config loading
schema registration
one-time validation
admin tools
CLI parsing
```

Avoid in:

```
per request middleware
tight loops
serialization hot path
packet decode path
metrics loop
```

# Better Optimization Strategies

## A. Use Concrete Types

Instead of:

```go
func Sum(items []interface{})
```

Use:

```go
func Sum(items []int)
```

Benefits:

```
no boxing
no type assertions
better cache locality
inlining possible
```

## Use Generics (Go 1.18+)

Instead of:

```go
func Max(a,b interface{})
```

Use:

```
func Max[T constraints.Ordered](a,b T) T
```

Compile-time polymorphism.

Often faster than interface dynamic dispatch.

## Cache Reflection Metadata

Bad:

```
for each request:
    parse struct tags
```

Good:

```
once at startup:
    cache field offsets/tags
```

Used by mature ORMs.

## Code Generation

Generate serializer:

```
protobuf
easyjson
sqlc
ent
```

Removes reflection at runtime.

## Use Type Switch Instead of Generic Reflection

Bad:

```
reflect.Kind()
```

Better:
```go
switch v := x.(type) {
case int:
case string:
}
```

Usually faster.

# How To Detect Problem

Use profiler:

```
go test -bench .
go test -benchmem
go tool pprof
```

Look for:
```
reflect.Value.Field
reflect.Value.Call
runtime.convT2E
runtime.assertI2T
mallocgc
```

These indicate interface/reflection overhead.

# Common Runtime Symbols
### convT2E
concrete type -> empty interface conversion

### convT2I
concrete type -> non-empty interface

### assertI2T
interface assertion to concrete type

If hot in pprof = investigate.
